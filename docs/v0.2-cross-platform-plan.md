# v0.2 — Cross-Platform Parity: Implementation Plan

> Status: **approved plan, pre-implementation.** Scope decisions are locked (see §Decisions).
> This document is grounded in the actual code at the time of writing (Swift, Kotlin, the
> `RuntimeBackend`/`BackendRuntime` streaming contract, `DownloadManager` resume, CI, ROADMAP).

## Goal

From v0.2 onward a developer writes their AI app **once** and runs it on **Android and iOS
(iPhone + iPad)** with **no change to JavaScript**. All platform differences stay behind the
`RuntimeBackend` abstraction. Feature parity — not implementation parity — is the objective.

---

## 0. Reframing: iOS is not greenfield

The SDK already ships working iOS native code. Both 0.1.0 and HEAD contain:

| Layer | Android | iOS | Status |
|---|---|---|---|
| File store / storage roots | `LocalAiFileStoreModule.kt` | `LocalAiFileStore.swift` (+`.m`) | Implemented both |
| Resumable download (Range, restart-on-200, cancel, throttled progress events) | Kotlin | Swift `URLSession` delegate | Implemented both |
| Streaming SHA-256 | Kotlin | Swift **CryptoKit** | Implemented both |
| Key-value store | Kotlin | `NSUserDefaults` | Implemented both |
| Device capabilities (RAM/CPU/storage/model id) | `LocalAiDeviceModule.kt` | `LocalAiDevice.swift` | Implemented both |
| Inference (load/stream/cancel/unload/benchmark) | via `llama.rn` | via `llama.rn` (already cross-platform) | Works both |
| Lifecycle logic (facade, DownloadManager + resume sessions, registry, BackendRuntime busy-guard/cancel policy, eligibility) | shared TypeScript | shared TypeScript | Platform-agnostic already |

The iOS Swift modules "mirror the Android contract exactly," and inference is delegated to
`llama.rn`'s own iOS build behind the `RuntimeBackend` seam — no per-platform engine code is
vendored.

**Consequence:** v0.2 parity is ~80% **validation, build-integration, CI, example-app, docs, and
performance** work, plus a small amount of **additive** JS API and Swift hardening — **not** a
from-scratch iOS runtime. The Swift code exists but there is no evidence it has been built/run on
a device; the real job is to **prove, harden, and CI-gate it.**

**Conflation corrected:** the "Swift + TurboModule + JSI + native streaming runtime" objective
describes the **future self-hosted `NativeLlamaCppBackend`** (the `cpp/` scaffold, explicitly "not
built", roadmap ~v0.6). It is a llama.cpp-via-JSI/JNI effort and is **not required for parity**,
because llama.rn already provides iOS inference behind the same abstraction. It is out of the v0.2
critical path.

## Decisions (locked)

1. **Canonical example:** the standalone public repo **`react-native-edge-slm-example`** gets the
   iOS project; its `src/` stays identical across platforms. The monorepo `apps/example` remains a
   JS snippet.
2. **Streaming API:** `generate()` becomes a **hybrid handle** — awaitable to `GenerateResult`
   **and** `AsyncIterable<{ text: string }>` (`for await`), with `onToken` retained. Early
   `break` auto-invokes `cancel()`. **Non-breaking, additive.**
3. **Native scope:** parity rides the existing **legacy-bridge Swift/Kotlin modules + `llama.rn`**
   (works under the New Architecture via interop). **No** TurboModule migration and **no** `cpp/`
   self-hosted runtime in v0.2 — deferred to v0.3+/v0.6.

## 1. Public API changes

All additive and non-breaking (honoring "don't change the public API unless necessary").

| Prompt/roadmap shows | Ships today | Change | Breaking? |
|---|---|---|---|
| `EdgeSLM.loadPreset(...)` | `LocalAI` singleton | Add `export const EdgeSLM = LocalAI` brand alias | No |
| `for await (const token of runtime.generate(...))` yielding `token.text` | `generate(): Promise<GenerateResult>` + `onToken(token: string)` | Hybrid handle: awaitable **and** async-iterable; keep `onToken` | No — **highest-risk item** |
| `getInstalledModels()` | absent (`getRegisteredPresets`/`getPresetStatus` only) | Add facade method reading the registry → `InstalledModel[]` | No |
| "Update model" | absent | Add `updatePreset(id)` (re-resolve source, re-download if checksum/size differs) — overlaps roadmap v0.3 "update detection" | No |
| "Resume downloads" (listed as a v0.2 goal) | **already implemented** (DownloadManager sessions + Swift/Kotlin Range resume) | Parity **validation** only | — |

Unchanged and already platform-agnostic: `registerPreset`, `configurePresetSource`,
`installPreset`, `loadPreset`, `cancel`, `unload`, `benchmark`.

### Async-iterable contract

`generate()` returns a `GenerationHandle` that is `PromiseLike<GenerateResult>` **and**
`AsyncIterable<{ text: string }>`. The async iterator's `return()` (fired on early `break`) calls
`runtime.cancel()`, preserving the single-flight busy-guard and existing cancel semantics.

## 2. Platform-specific challenges

1. **CI native builds are new.** Current CI is one Ubuntu job (typecheck/test/build,
   `--ignore-scripts`). Adding Android + iOS builds means a **macOS runner**, CocoaPods,
   `xcodebuild`, and running `llama.rn`'s native-artifact postinstall (dropping `--ignore-scripts`
   there). Split a light always-on `verify` job from gated, cached, build-only native jobs.
2. **iOS background downloads.** The Swift downloader uses a foreground `.default` `URLSession` +
   a **blocking semaphore per download**. App suspension cancels it and the pattern doesn't scale.
   Foreground parity is fine for v0.2; true background downloads (`URLSessionConfiguration.background`
   / `BGTaskScheduler`) belong to v0.3. Flag, don't fix now.
3. **Memory-pressure handling.** iOS jetsams the app; we can't intercept the llama.rn context's
   native allocation. v0.2 scope: subscribe to `didReceiveMemoryWarningNotification` and expose an
   advisory hook / eligibility gating that can proactively `unload()`. True in-engine handling
   needs llama.rn support or the native backend — document the limitation.
4. **Metal / acceleration.** llama.rn enables Metal on iOS device builds already; nothing to
   implement for parity. Core ML / NNAPI / Vulkan are future-backend concerns (v0.6). Simulator
   lacks a real GPU — perf numbers must come from physical devices.
5. **TurboModule/JSI.** Existing modules are legacy bridge (`RCT_EXTERN_MODULE`, `RCTEventEmitter`,
   `NativeModules`); they work under the New Architecture via interop. Full codegen-TurboModule
   migration is real cross-platform work and **not required for parity** — deferred.
6. **Storage/RAM semantics differ** (`volumeAvailableCapacityForImportantUsage` vs `StatFs`;
   App Support vs filesDir). Already handled — document in the matrix / perf report.

## 3. Architectural risks

| Risk | Severity | Mitigation |
|---|---|---|
| Async-iterable + busy-guard leak (early `break` without cancel → native decode keeps running) | High | Implement `AsyncIterator.return()` → `cancel()`; cover with tests |
| CI cost/velocity (real Android+iOS builds on every PR throttle the PR workflow) | Medium | Gate + cache native jobs; keep `verify` fast and always-on |
| Two diverging example apps (standalone full app vs monorepo JS snippet) | Medium | Standalone repo is canonical; keep `apps/example` a documented snippet |
| Runtime-agnostic principle vs `llama.rn` coupling | Medium | No iOS special-casing in JS; keep everything behind `RuntimeBackend` |
| ROADMAP vs reality drift (e.g. resume already implemented but listed under v0.3) | Low | Refresh the roadmap as part of v0.2 docs |

## 4. Phased delivery (each a PR)

1. **Prove iOS end-to-end** — add iOS to the canonical example, `pod install`, build, run the full
   lifecycle on Simulator + one real device. *(In progress — see example-repo PR.)*
2. **Harden Swift** to match Android edge cases surfaced in Phase 1 (error mapping, cancel races,
   event throttling).
3. **Additive API** — `EdgeSLM` alias, `getInstalledModels()`, async-iterable `generate()`
   (+ `return()`→cancel), optional `updatePreset()`. Unit-tested via injected-env fakes.
4. **CI expansion** — split verify / build-android / build-ios / example-build jobs; add a
   **public-API-surface snapshot** (e.g. api-extractor) to guarantee cross-platform contract
   identity.
5. **Parity conformance suite** — one shared lifecycle test driven on both platforms (device
   self-test screen or Detox/Maestro): install → verify → load → stream → cancel → benchmark →
   delete.
6. **Docs + compatibility matrix + migration guide** (migration is minimal — additive changes).
7. **Performance report** — methodology + numbers on ≥1 real Android and ≥1 real iOS device
   (model loading, first-token latency, tokens/sec, peak memory, download speed, storage usage).

| Capability | Android | iOS |
|---|:---:|:---:|
| Download / resume / checksum | ✅ | ✅ |
| Registry / delete / update | ✅ | ✅ |
| GGUF via llama.cpp | ✅ | ✅ |
| Load / unload | ✅ | ✅ |
| Streaming | ✅ | ✅ |
| Cancellation | ✅ | ✅ |
| Benchmark | ✅ | ✅ |
| Native acceleration | NNAPI / Vulkan / GPU | Metal |

Feature parity is guaranteed; native acceleration and background behavior may differ per platform
and remain hidden behind `RuntimeBackend`.

## Success criteria

Clone the example → run on Android → run the **same** project on iOS → install the same model →
stream responses → benchmark → **using the same JavaScript on both platforms.**
